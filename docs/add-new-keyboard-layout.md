# Add a New Keyboard Layout

This guide is for adding a completely new keyboard to this repo when it does not match the existing Go60, Glove80, or SliceMK geometry.

Use this only when you are integrating a new board.

If you just want to make a normal layout change, use [getting-started.md](../getting-started.md) instead.

The important constraint is that you are not creating a fourth independent keymap. You are plugging a new board into the same shared keymap library and sync model.

If you violate that assumption, the repo stops being modular and the maintenance cost rises immediately.

---

## The model

Go60 remains the source of truth for shared layer content. Everything sources from the go60 layout.

That means:

- shared layer edits start in `boards/go60/layers/`
- shared behaviors, macros, timings, combos, and layer constants stay in `shared/`
- your new board receives shared bindings through `scripts/keymapsync.sh`
- keys that exist uniquely on any new board are maintained directly in that board's layer files

Think of a new board as two things at once:

1. a translated view of the Go60 shared layout for the 60 positions the boards have in common
2. a board-local extension for any extra keys, fork-specific exclusions, or hardware-specific behavior

---

## What sync will and will not do

`scripts/keymapsync.sh` copies Go60 bindings into a target board using a translation map.

It does this per layer:

1. read the Go60 `bindings = < ... >;` block
2. read the target board's bindings block
3. replace only the positions named in `boards/translations/go60_to_<board>.map`
4. leave all non-mapped target positions untouched

That last point is the one that matters most for a completely different board.

If your new board has extra columns, an F-row, macro keys, encoder buttons, or extra thumbs, those keys are not owned by Go60. They are owned by the target board and must be maintained in the target board's layer files.

Practical rule:

- mapped shared positions are regenerated by sync
- unmapped board-only positions are maintained by hand on the target board

Do not edit mapped shared positions directly in the target board. The next sync will overwrite them.

## Before you start

Decide these three things first:

### 1. Which 60 logical positions are shared with Go60

The sync model assumes the new board can express the same 60 logical shared positions defined by Go60.

Those positions use names like:

- `POS_LH_C6R1`
- `POS_LH_C5R3`
- `POS_RH_C1R2`
- `POS_LH_T1`
- `POS_RH_T3`

If your board cannot express those shared positions in a meaningful way, it may not fit this repo's architecture.

### 2. Which keys are board-only extras

These are keys that exist on the new board but not on Go60, such as:

- inner columns
- outer columns
- top macro row
- encoder press positions
- extra thumb keys

These keys will live directly in `boards/<board>/layers/*.dtsi` and must be maintained there.

### 3. Which ZMK fork and build path the board needs

Choose whether the board will build like:

- Go60 and Glove80 via Nix and `moergo-sc/zmk`
- SliceMK via west and a different fork

If the board needs fork-specific exclusions, document them immediately and keep them local to that board.

- `docs/keyPositionMapping.md`: records the board's logical-to-physical mapping for future map maintenance

If the new board follows the Go60 or Glove80 build path, also create or update:

- `build/<board>.nix` create
- `.github/workflows/build.yml` update with a Nix build job

This is the path for boards that build against `moergo-sc/zmk` or a similar Nix-compatible fork.

### D. Conditional files for a west-based board

If the new board follows the SliceMK-style west path, you will usually need to create or update:

- `config/<entrypoint>.keymap` create as a thin wrapper that includes `../boards/<board>/<board>.keymap`
- `config/<shield-or-board>.conf` create or update for the board's west build config
- `build.yaml` update if the build matrix is used for this board family
- `config/west.yml` update if the west dependency source changes
- `.github/workflows/build.yml` update with the correct west build invocation

This is the path for boards that do not build through the existing Nix derivations.

### E. Optional but strongly recommended updates

These are not always required for the build to succeed, but should usually be updated during a real integration:

- `docs/add-new-keyboard-layout.md`
- `docs/completedplans/repo-assessment.md` if the architecture meaningfully changes
- any board-specific RCA or notes file if the new board needs special exclusions

If the repo can build the new board but the docs still describe only the old three-board world, the integration is not complete.

## Step 1: Create `positions.dtsi`

This file is the contract between the shared logical layout and the board's physical key numbering.

Define all 60 shared Go60 logical positions first. Then define any board-only positions.

Shared positions must use the same logical names as Go60.

Example shape:

```c
#define POS_LH_C6R1   <physical index>
#define POS_LH_C5R1   <physical index>
...
#define POS_RH_T3     <physical index>

// Board-only extras
#define POS_LH_C0R1   <physical index>
#define POS_LH_C0R2   <physical index>
#define POS_LH_F1     <physical index>
```

Rules:

- use Go60 logical names for every shared position
- assign physical indices in the exact order the target board layer bindings will use
- keep board-only names distinct and local to the board

If this file is wrong, everything else will look wrong later.

---

## Step 2: Create `position_groups.dtsi`

This file tells shared HRM logic which keys belong to each hand and which positions can trigger opposite-hand holds.

You need these macros:

- `LEFT_HAND_KEYS`
- `RIGHT_HAND_KEYS`
- `THUMB_KEYS`
- `HRM_LEFT_TRIGGER_POSITIONS`
- `HRM_RIGHT_TRIGGER_POSITIONS`

The safest starting point is to copy an existing board and then replace the key lists with your board's logical positions.

Be careful here:

- hand groups should include board-only keys if they belong to that hand
- trigger groups should match how you want opposite-hand HRM activation to behave

If the board shape is unusual, this file is where the behavior usually needs deliberate tuning.

---

## Step 3: Create `board_meta.dtsi`

This file is for board-local helpers and hardware-specific behavior.

Keep these concerns here:

- tap-dance layer helpers tied to this board
- pointing device specifics
- RGB-specific helpers
- anything fork-specific that should not leak into shared code

For a board with minimal custom hardware, copy the smallest existing board meta file and keep it thin.

---

## Step 4: Create `<board>.keymap`

Use `boards/go60/go60.keymap` as the canonical pattern for include ordering and wrapper placement.

You must preserve these structure rules:

1. board-local includes for `positions.dtsi` and `position_groups.dtsi`
2. `../../shared/layers.dtsi`
3. `../../shared/global_timings.dtsi`
4. `board_meta.dtsi`
5. shared behavior and macro includes inside `/ { behaviors { ... } }`
6. combo includes inside `/ { combos { compatible = "zmk,combos"; ... } }`
7. layer includes inside `/ { keymap { compatible = "zmk,keymap"; ... } }`

Decide here whether the board includes all 21 layers or intentionally excludes one, as SliceMK excludes `magic.dtsi`.

Also decide here which ZMK headers belong to this fork. For example, some forks need `zmk/pointing.h`, while others use `zmk/mouse.h`.

---

## Step 5: Create `<board>.conf`

Put board-specific build configuration here.

This file should contain only the settings specific to this board or shield. Shared logic belongs elsewhere.

---

## Step 6: Create all layer files

Start by copying Go60's layer files.

```sh
cp boards/go60/layers/*.dtsi boards/<board>/layers/
```

Then adapt every file so the binding count matches the new board's physical key count.

This is where the board becomes "completely different" in a controlled way.

For each layer file:

1. keep the Go60-derived shared bindings for the 60 shared positions
2. add bindings for the new board-only positions
3. use `&trans` for extra keys that should inherit behavior
4. use explicit board-only bindings when those extra keys should do real work

The critical maintenance rule is this:

- shared mapped positions should stay conceptually aligned to Go60
- board-only positions are owned by the target board and are expected to differ

---

## Step 7: Create the translation map

Create `boards/translations/go60_to_<board>.map`.

Each line is:

```text
<go60 index> <target index>
```

This file should contain exactly the shared positions that Go60 and the new board have in common.

For most boards in this repo, that means 60 entries.

Example:

```text
0 10
1 11
2 12
...
54 69
```

Build the map from logical names, not from visual guesses.

Recommended method:

1. open `boards/go60/positions.dtsi`
2. open `boards/<board>/positions.dtsi`
3. match identical logical `POS_*` names
4. write the Go60 physical index and target physical index pair

The translation map is the sync boundary. If a key is not in the map, sync will not manage it.

That is exactly how board-only keys stay board-local.

### How translation maps are actually made

There is no separate map generator in this repo right now. The maps are made by matching identical logical `POS_*` names between:

- `boards/go60/positions.dtsi`
- `boards/<board>/positions.dtsi`

Then writing the physical index pair for each shared logical position.

The process is manual, but it is deterministic.

### Exact procedure

1. Open `boards/go60/positions.dtsi`.
2. Open `boards/<board>/positions.dtsi`.
3. List every logical `POS_*` name that exists in both files.
4. For each shared name, take the numeric value from Go60 as the first column.
5. Take the numeric value from the target board as the second column.
6. Write one line per shared position as `<go60 index> <target index>`.
7. Sort the final file by the Go60 index ascending.

That means the map is derived from logical identity, not from physical row order, visual symmetry, or screen position.

### Worked example: Go60 to Glove80

In Go60:

```c
#define POS_LH_C6R1 0
#define POS_RH_C1R4 42
#define POS_LH_T1   54
```

In Glove80:

```c
#define POS_LH_C6R1 10
#define POS_RH_C1R4 58
#define POS_LH_T1   69
```

So the map lines are:

```text
0 10
42 58
54 69
```

That is exactly what you see in `boards/translations/go60_to_glove80.map`.

### Worked example: Go60 to SliceMK

In Go60:

```c
#define POS_LH_C6R1 0
#define POS_RH_C1R1 6
#define POS_RH_C1R4 42
```

In SliceMK:

```c
#define POS_LH_C6R1 1
#define POS_RH_C1R1 9
#define POS_RH_C1R4 49
```

So the map lines are:

```text
0 1
6 9
42 49
```

That matches `boards/translations/go60_to_slicemk.map`.

### What does not go into the map

Board-only keys do not get entries.

Examples from the current repo:

- Glove80 function-row keys like `POS_LH_C5R0`
- Glove80 extra bottom-row keys like `POS_LH_C1R5`
- SliceMK inner-column keys like `POS_LH_C0R1`
- SliceMK special top key `POS_SPECIAL_R0`
- SliceMK extra thumbs like `POS_LH_T4`, `POS_LH_T5`, `POS_LH_T6`

These positions have no Go60 equivalent, so they must not appear in `go60_to_<board>.map`.

If you add them to the map anyway, sync will start treating them like shared keys and overwrite them from Go60, which is wrong.

### Sanity checks for a finished map

Before using the map, verify all of these:

- every line is two integers separated by whitespace
- the first column is a Go60 index
- the second column is the target board index for the same logical `POS_*` name
- the file is sorted by the first column ascending
- no shared logical position is missing
- no board-only key appears in the file
- no Go60 index is repeated
- no target index is repeated unless you intentionally want two Go60 positions to collide, which this repo does not do

### Practical workflow for building a new map

The easiest low-error workflow is:

1. copy all `#define POS_...` lines from Go60 into a scratch buffer
2. copy all `#define POS_...` lines from the new board into another scratch buffer
3. match by identical logical names
4. write the numeric pairs into `boards/translations/go60_to_<board>.map`
5. compare the first and last several entries against the positions files by hand
6. run `bash scripts/keymapsync.sh`
7. inspect one or two synced layers to confirm shared keys landed where expected and board-only keys were preserved

If the synced layer looks wrong, the map is usually wrong.

## Step 8: Wire the board into `scripts/keymapsync.sh`

Add the new board as a target alongside Glove80 and SliceMK.

You need:

- a target layer directory variable
- a loaded forward map for `go60_to_<board>.map`
- a sync loop that calls `sync_layer`

Conceptually the new code should do the same thing as the existing targets:

```bash
NEWBOARD_DIR="$REPO_ROOT/boards/<board>/layers"

declare -A fwd_newboard
load_map "$TRANS_DIR/go60_to_<board>.map" fwd_newboard

for f in "$GO60_DIR"/*.dtsi; do
  sync_layer "$f" "$NEWBOARD_DIR/$(basename "$f")" fwd_newboard <board>
done
```

Important behavior inherited from `sync_layer`:

- mapped positions are replaced from Go60
- unmapped positions are preserved
- surrounding comments and most spacing are preserved

If the new board has a fork-specific exclusion like SliceMK's `&magic` rewrite, add that logic here and keep it isolated to that board.

## Step 9: Wire the board into `scripts/validation.sh`

Validation is not optional for a new board. Add it everywhere the existing boards are checked.

At minimum, extend these areas:

- required files
- layer file presence
- keymap layer include count
- position count
- binding count per layer
- combo wrapper presence
- shared include checks
- include ordering checks
- board-specific header or fork exclusions
- cross-board layer and reference checks

Use the existing board cases as the pattern. Do not add a board without validation coverage.

## How board-only keys are added and maintained

This is the part that usually gets misunderstood.

### Board-only keys live in target layer files

Examples:

- Glove80 F-row
- SliceMK inner columns
- extra thumbs on a future board
- macro cluster keys on a future board

These keys do not belong in Go60 because Go60 cannot represent them.

They should be added directly to:

- `boards/<board>/layers/base.dtsi`
- `boards/<board>/layers/typing.dtsi`
- and the rest of that board's layer files as needed

### Sync will preserve them if they are not mapped

This is the maintenance contract:

- if a board-only key is absent from `go60_to_<board>.map`, sync leaves it alone
- if you accidentally put a board-only key into the map, sync starts treating it like a shared key and will overwrite it from Go60

So the map is not just a translation artifact. It is also the ownership boundary.

### Recommended workflow for changes touching both shared and board-only keys

1. edit the shared change in `boards/go60/layers/<layer>.dtsi`
2. run `bash scripts/keymapsync.sh`
3. edit `boards/<board>/layers/<layer>.dtsi` for any board-only positions
4. run `bash scripts/validation.sh`
5. commit the synchronized set together

### Recommended defaults for board-only keys

Use these intentionally:

- `&trans` when the extra key should inherit lower-layer behavior
- `&none` when the extra key should do nothing and block fallthrough
- a real binding when the extra key is part of the board's unique UX

## How future shared changes behave on the new board

After integration, day-to-day maintenance should work like this:

### Shared behavior change

Example: a symbol key moves on Base.

Result:

- edit Go60 once
- sync propagates that mapped position into the new board
- board-only keys on the new board remain unchanged

### New board-only key behavior

Example: you add a macro row action only on the new board.

Result:

- edit only the target board's layer files
- do not touch Go60
- do not add that key to the translation map

### Shared new layer or removed layer

Result:

- update `shared/layers.dtsi`
- add or remove layer files for every board, including the new one
- update each board keymap include list
- update validation expectations

### Shared behavior file change

Example: new macro or timing constant.

Result:

- edit `shared/`
- no translation map change required
- validate that the new board still compiles under its fork

## Fork-specific exceptions

If the new board uses a fork with missing or incompatible features, isolate the exception in the new board's local surfaces.

Typical places:

- `<board>.keymap` include list
- `board_meta.dtsi`
- `scripts/keymapsync.sh` target-specific rewrite rule
- `scripts/validation.sh` target-specific exclusion checks

Do not weaken shared files unless every board really needs the change.

SliceMK is the working example of this pattern.

## Exact bring-up checklist

Use this order.

1. Create `boards/<board>/` and `boards/<board>/layers/`.
2. Add `positions.dtsi` with all shared `POS_*` names plus board-only extras.
3. Add `position_groups.dtsi`.
4. Add `board_meta.dtsi`.
5. Add `<board>.keymap` with correct include ordering and wrapper blocks.
6. Add `<board>.conf`.
7. Copy all Go60 layer files into `boards/<board>/layers/`.
8. Expand every layer to the new board's binding count.
9. Set board-only keys to `&trans`, `&none`, or real board-local bindings.
10. Create `boards/translations/go60_to_<board>.map`.
11. Extend `scripts/keymapsync.sh`.
12. Extend `scripts/validation.sh`.
13. Add build wiring.
14. Run `bash scripts/validation.sh` and fix structural failures.
15. Run `bash scripts/keymapsync.sh`.
16. Re-check that board-only positions stayed intact after sync.
17. Run `bash scripts/validation.sh` again.
18. Build the board through its real fork path.
19. Document any board-specific exclusions.

## Failure modes to watch for

### Shared positions edited directly on the new board

Symptom:

- your changes disappear after sync

Cause:

- you edited a mapped position owned by Go60

Fix:

- move the shared change back to Go60 and sync again

### Board-only key accidentally added to the translation map

Symptom:

- sync starts overwriting a key that should stay board-local

Fix:

- remove that entry from `go60_to_<board>.map`

### Layer binding counts stop matching the board

Symptom:

- validation fails on per-layer binding counts

Fix:

- re-check `positions.dtsi` and every layer's `bindings` block length

### Fork-specific includes copied blindly from another board

Symptom:

- build fails on missing headers or unsupported features

Fix:

- reduce the board to the minimal correct keymap includes for its fork
- isolate exceptions locally instead of weakening shared code
